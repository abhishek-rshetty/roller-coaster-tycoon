# Browser Tycoon V1.1 Implementation Spec

## Summary

This document translates `V1.1_SPEC.md` into implementation-ready changes for the current React + TypeScript MVP. The work keeps the game fully local, preserves the monthly strategy loop, and extends the existing state model, simulation output, and UI to support Indianized presentation, planned builds, historical reports, and per-ride performance reporting.

The implementation should be done in behavior-first order:

1. rename and type migration
2. reporting/data model upgrades
3. UX clarity improvements
4. planned-build state flow
5. Indianization and rebalancing
6. ride-level report output

## 1. Current Baseline

Current implementation facts:

- `RideDefinition` uses `footprint`
- `GameState` uses `landUsed`, `landCapacity`, and a single `rides` array
- build actions make rides active immediately
- `MonthlyReport` only stores park-level totals plus messages
- report UI only shows `latestReport`
- currency is US-style `$`
- ride cards expose raw stats without explanatory help text

V1.1 must preserve existing playable behavior while replacing these structures cleanly.

## 3. Action Layer Changes

### 3.1 `createNewGame`

In `src/game/actions.ts`:

- initialize the new `GameState` shape
- no active rides
- no planned rides
- no reserved values

### 3.2 `buildRide`

Change semantics from “activate immediately” to “queue build”.

New behavior:

- lookup ride by `rideId`
- compute `availableCash = cash - reservedCash`
- compute `availableArea = areaCapacity - areaUsed - reservedArea`
- reject if `availableCash < buildCost`
- reject if `availableArea < areaRequired`
- append to `plannedRides`
- increment `reservedCash` by `buildCost`
- increment `reservedArea` by `areaRequired`
- do not change `cash`
- do not change `areaUsed`
- do not add to `activeRides`

Error copy:

- `Insufficient cash.` stays acceptable
- `Insufficient land.` must become `Not enough area available.`

### 3.3 Add `cancelPlannedRide(planId)`

Add a new action:

```ts
cancelPlannedRide(gameState: GameState, planId: string): GameState
```

Behavior:

- find the planned ride
- remove it from `plannedRides`
- decrement `reservedCash` by `buildCost`
- decrement `reservedArea` by `areaRequired`
- throw if no matching planned ride exists

### 3.4 `simulateNextMonth`

Change sequencing carefully.

Required order:

1. simulate the month using `activeRides` only
2. build the `MonthlyReport`
3. update cash, satisfaction, reputation, market growth
4. convert all current `plannedRides` into `activeRides`
5. increment `areaUsed` by total planned ride area
6. clear `plannedRides`
7. reset `reservedCash` to `0`
8. reset `reservedArea` to `0`
9. append `latestReport` to `monthlyHistory`
10. increment month

Important behavior:

- newly planned rides must not affect the month being simulated
- they become active for the following month

### 3.5 Storage compatibility

In `src/game/storage.ts`:

- continue using the same save key
- add a lightweight migration path for old saves
- if an older save is loaded:
  - map `rides` -> `activeRides`
  - map `landUsed` -> `areaUsed`
  - map `landCapacity` -> `areaCapacity`
  - initialize `plannedRides`, `reservedCash`, and `reservedArea`
- if migration cannot be completed safely, return `null`

## 4. Simulation and Report Changes

### 4.1 Use `activeRides` everywhere

In `src/game/simulation.ts`, all attraction, capacity, maintenance, and market-fit logic must operate on `activeRides`.

Replace all current `state.rides` usage with `state.activeRides`.

### 4.2 Add in-park revenue breakdown fields

While building the simulation snapshot and monthly report, persist:

- `averageGuestSpend = 8 * spendingPower * satisfaction`
- `marketSpendingModifier = selectedLocation.spendingPower`
- `satisfactionRevenueModifier = satisfaction`

These values must be stored in `MonthlyReport` so historical month selection can display exact past values.

### 4.3 Add per-ride performance estimation

Implement a helper in `src/game/simulation.ts`:

```ts
buildRidePerformanceReport(state: GameState, visitors: number): RidePerformanceReport[]
```

Required method:

1. build the same per-ride market fit and duplicate-adjusted attraction values already used for total attraction
2. compute `attractionShare` for each active ride:
   - `rideAdjustedAttraction / totalAdjustedAttraction`
3. estimate visitors:
   - `estimatedVisitors = Math.floor(visitors * attractionShare)`
4. compute utilization:
   - `utilizationRate = clamp(estimatedVisitors / monthlyCapacity, 0, 1)`
5. compute `marketFitScore = marketFit / 5`
6. store `maintenanceCost = ride.monthlyMaintenance`
7. assign one deterministic comment using the rule priority below

Comment priority:

1. duplicate multiplier below `1`
   - `Value is reduced because similar rides are already in the park.`
2. utilization >= `0.9`
   - `This ride is close to capacity and may benefit from support expansion.`
3. market fit score >= `0.75` and utilization >= `0.6`
   - `Strong fit for this market and performing well.`
4. monthly maintenance high relative to estimated visitors
   - threshold: `monthlyMaintenance / max(estimatedVisitors, 1) > 2`
   - comment: `Expensive to maintain for its current demand.`
5. market fit score <= `0.45`
   - `This ride is not a strong match for your current market.`
6. fallback
   - `Steady contributor this month.`
