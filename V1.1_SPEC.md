# Browser Tycoon V1.1 Spec

## Summary

V1.1 improves clarity, local relevance, and decision-making. It does not change the core monthly strategy loop, but it adds better explanations, historical reporting, Indianized market framing, pre-month ride cancellation, and estimated per-ride performance reporting.

## Goals

- Make the game feel Indian in setting, pricing language, and market context.
- Remove ambiguous labels like `footprint`.
- Make build stats self-explanatory.
- Let players undo build decisions before ending the month.
- Fix monthly report readability.
- Let players inspect previous months.
- Explain in-park revenue clearly.
- Add estimated last-month performance for each ride.

## Non-Goals

- No backend
- No multiplayer
- No map or construction mode
- No guest-level pathfinding
- No exact ride-by-ride guest simulation
- No weather, staff, shops, or breakdowns

## 4. Planned Builds and Cancel Before Month End

### 4.1 New Behavior

Building a ride should no longer make it active immediately.

Instead:
- building creates a `planned ride`
- planned rides reserve cash and area immediately
- planned rides do not contribute to attraction, capacity, maintenance, or revenue until next month is run
- player may cancel planned rides any time before `Run Next Month`

### 4.2 New State

Add:
```ts
type PlannedRide = RideDefinition & {
  planId: string;
  plannedMonth: number;
};

type GameState = {
  ...
  activeRides: BuiltRide[];
  plannedRides: PlannedRide[];
  areaUsed: number;
  areaCapacity: number;
  reservedCash: number;
  reservedArea: number;
};
```

### 4.3 Build Behavior

When player clicks `Build`:
- validate against available cash after reservations
- validate against available area after reservations
- add ride to `plannedRides`
- increase `reservedCash`
- increase `reservedArea`

### 4.4 Cancel Behavior

Each planned ride row must have a `Cancel` button.

When player clicks `Cancel`:
- remove ride from `plannedRides`
- decrease `reservedCash`
- decrease `reservedArea`

### 4.5 Month Transition

When player runs next month:
- simulate only with currently active rides
- after the month simulation completes, all planned rides convert into active rides
- reserved values are cleared for converted rides
- those rides begin contributing from the following month onward

### 4.6 UI

Add a new section:
- `Planned Builds`

Each row shows:
- ride name
- category
- area
- build cost
- cancel button

## 8. Per-Ride Last Month Performance

### 8.1 Scope

This is an estimated attribution system, not exact guest-level simulation.

### 8.2 Goal

For the selected report month, show how each active ride performed.

### 8.3 Required Per-Ride Fields

Add ride performance records into `MonthlyReport`:
```ts
type RidePerformanceReport = {
  instanceId: string;
  name: string;
  category: RideCategory;
  estimatedVisitors: number;
  utilizationRate: number;
  attractionContribution: number;
  marketFitScore: number;
  maintenanceCost: number;
  comment: string;
};

type MonthlyReport = {
  ...
  ridePerformance: RidePerformanceReport[];
};
```

### 8.4 Estimation Model

For each active ride in the simulated month:
- compute adjusted attraction contribution
- compute each ride’s share of total attraction
- allocate estimated visitors using attraction share
- cap effective ride usage relative to ride capacity
- compute utilization:

```ts
utilizationRate = estimatedVisitors / monthlyCapacity
```

Clamp utilization to `0..1`.

### 8.5 Comments

Each ride gets one deterministic comment.

Rules:
- high market fit + high utilization:
  - `Strong fit for this market and performing well.`
- high maintenance + low utilization:
  - `Expensive to maintain for its current demand.`
- duplicate-penalized ride:
  - `Value is reduced because similar rides are already in the park.`
- low market fit:
  - `This ride is not a strong match for your current market.`
- near full capacity:
  - `This ride is close to capacity and may benefit from support expansion.`

### 8.6 UI

In the report panel, below monthly summary, add:
- `Ride Performance`

For the selected month, show a table or list with:
- Ride
- Category
- Estimated Visitors
- Utilization
- Attraction Contribution
- Market Fit
- Maintenance
- Comment

## 10. Action Changes

### createNewGame

- initialize `activeRides` as empty
- initialize `plannedRides` as empty
- initialize `reservedCash` and `reservedArea` as `0`

### buildRide

- add to `plannedRides`
- reserve cash and area
- do not add to active rides yet

### cancelPlannedRide(planId)

- remove from `plannedRides`
- release reserved cash and area

### simulateNextMonth

- simulate only using `activeRides`
- after simulation, convert all current planned rides into active rides
- clear reservations

- Selected report month shows per-ride estimated performance
- Each ride has a deterministic comment
- Data is available for all active rides in that month
